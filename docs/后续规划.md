## 阶段二：Java 爬虫数据采集（Day33 ~ Day38）

### 整体目标

```
建立"热点资讯知识库"，让 RAG 系统能回答：
  - "今天 GitHub 最火的 Java 项目是什么？"
  - "小红书最近热门的 AI 工具有哪些？"
  - "抖音热门话题排行"
  - "微博实时热搜前 10"

数据流：爬虫 --> 清洗 --> 向量化 --> 入库 --> RAG 问答
```

### 技术选型

```
框架对比（Java 生态）：

  WebMagic       --> 国产爬虫框架，基于 HttpClient + Jsoup，文档中文
                    优点：上手快，官方有微博/知乎案例
                    缺点：维护一般，反爬能力弱

  Selenium       --> 浏览器自动化，能跑 JS
                    优点：能爬 SPA 网站（小红书、抖音）
                    缺点：重量级，慢，吃内存

  Playwright Java --> Selenium 替代品，更现代
                    优点：支持多浏览器，API 简洁，反爬能力强
                    缺点：Java SDK 不如 Python 成熟

  Jsoup          --> 纯 HTML 解析，不是爬虫
                    适合静态页面（如微博热搜榜）

推荐组合：
  静态页面（微博热搜、GitHub Trending）  --> Jsoup + OkHttp
  动态 SPA（小红书、抖音）              --> Playwright Java
  调度框架                              --> XXL-Job 或 Spring @Scheduled
```

## 阶段四：AI SQL 智能助手（Day42 ~ Day48）

### 1. 需求与场景

```
目标：自然语言 --> 标准 SQL --> 安全执行 --> 结果可视化

典型场景：
  用户："查一下上周销售额 Top 10 的商品"
       ↓
  系统：自动生成 SELECT product_id, SUM(amount) FROM orders
              WHERE created_at >= ... GROUP BY ... ORDER BY ... LIMIT 10
       ↓
  执行 + 表格/图表展示

价值：
  - 业务人员不用学 SQL 也能查数据
  - DBA 不用写一次性 SQL（每天被业务方 @ 烦了）
  - 取代部分 BI 工具（Tableau / FineBI）
```

### 2. 整体技术架构

```
┌──────────┐   自然语言    ┌─────────────────┐   SQL    ┌──────────┐
│ 前端 Vue │ ──────────→  │ AI SQL Service  │ ──────→ │ 业务库   │
└──────────┘              └─────────────────┘          └──────────┘
                          ↑    ↑     ↑     ↑
                          │    │     │     │
                  Schema RAG  限流  缓存  审计
                  (pgvector) (Sentinel)(Caffeine+Redis)(Kafka)

核心流程：
  1. 用户输入自然语言
  2. RAG 检索：查相关表结构（DDL）+ 历史 SQL 案例
  3. LLM 生成 SQL（带 Few-Shot 示例）
  4. SQL 校验：JSqlParser 解析 + 黑名单检查 + 权限校验
  5. 执行：Druid 数据源（只读账号 + SQL 防火墙）
  6. 结果缓存（Caffeine 一级 + Redis 二级）
  7. 返回前端：表格 / 图表 / 导出 Excel
```

#### Day43 NL → SQL 生成核心

```
Prompt 设计（Few-Shot + Schema RAG）：

  你是 SQL 专家。根据用户问题生成 PostgreSQL 标准 SQL。

  【相关表结构】
  {从 RAG 检索到的 Schema 描述}

  【历史相似查询】
  问：上月订单总数
  SQL：SELECT COUNT(*) FROM orders WHERE created_at >= ...

  【约束】
  - 只能 SELECT，不能 INSERT/UPDATE/DELETE/DROP/ALTER
  - 必须 LIMIT，默认 100
  - 时间字段使用 created_at
  - 返回 JSON：{"sql": "...", "explanation": "..."}

  【用户问题】
  {question}

实现：
  @Service
  public class NlToSqlService {
      public SqlGenResult generate(String question) {
          // 1. RAG 检索相关表
          List<Document> tables = vectorStore.similaritySearch(
              SearchRequest.builder()
                  .query(question)
                  .topK(5)
                  .filterExpression(b.eq("type", "schema").build())
                  .build()
          );

          // 2. 检索历史成功案例（提升质量）
          List<Document> samples = vectorStore.similaritySearch(...);

          // 3. 拼 Prompt
          String prompt = template.render(question, tables, samples);

          // 4. 调 LLM
          String json = chatClient.prompt(prompt).call().content();

          // 5. 解析返回
          return parseResult(json);
      }
  }

输出对象：
  class SqlGenResult {
      String sql;             // 生成的 SQL
      String explanation;     // 解释（中文）
      List<String> tables;    // 涉及的表
      Double confidence;      // 置信度 0~1
  }
```

#### Day44 SQL 校验（安全核心 ★★★）

```
四层防御：

第一层：JSqlParser 语法校验
  Statement stmt = CCJSqlParserUtil.parse(sql);
  if (!(stmt instanceof Select)) {
      throw new SqlForbiddenException("仅允许 SELECT");
  }

第二层：AST 危险操作检查
  - 不能有子查询里嵌 INSERT/UPDATE/DELETE
  - 不能用 SELECT INTO（变相写表）
  - 不能用 pg_catalog.pg_xxx 系统函数（可能泄露密码哈希）
  - 不能用 COPY / EXECUTE / DO

第三层：表权限校验
  从 SQL 中提取所有引用的表（TablesNamesFinder）
  对照当前用户允许访问的表白名单
  不在白名单 --> 拒绝

第四层：执行限制（运行时）
  - SET statement_timeout = '5s'（PG 强制超时）
  - SET LOCAL work_mem = '64MB'（防大查询打爆内存）
  - LIMIT 强制注入（如果用户/LLM 没写 LIMIT，自动加 LIMIT 1000）
  - 单次结果集 row count > 10000 直接截断

代码草图：
  @Component
  public class SqlValidator {
      private static final Set<Class<?>> ALLOWED = Set.of(Select.class);
      private static final Set<String> FORBIDDEN_FUNCS = Set.of(
          "pg_read_file", "pg_ls_dir", "pg_stat_file"
      );

      public void validate(String sql, String username) {
          Statement stmt = CCJSqlParserUtil.parse(sql);

          // 第一层
          if (!ALLOWED.contains(stmt.getClass())) {
              throw new SqlForbiddenException("不允许的 SQL 类型");
          }

          // 第二层
          checkForbiddenFunctions(stmt);

          // 第三层
          List<String> tables = new TablesNamesFinder().getTableList(stmt);
          checkTablePermission(username, tables);

          // 第四层（在执行时由 JdbcTemplate / Druid 处理）
      }
  }

Druid SQL 防火墙双保险：
  spring:
    datasource:
      druid:
        filter:
          wall:
            enabled: true
            config:
              select-allow: true
              delete-allow: false
              update-allow: false
              drop-table-allow: false
              alter-table-allow: false
              truncate-allow: false
```

#### Day45 高并发执行架构

```
高并发挑战：
  - 100 用户同时问 SQL --> 100 次 LLM 调用 + 100 次 DB 查询
  - LLM 慢（3~5s/次）+ 复杂 SQL 慢（1~10s/次）= 单请求 10s+
  - 不优化必死

四层优化：

层1：Caffeine 本地缓存（最快，单 JVM）
  Key：md5(question)
  TTL：5 分钟
  容量：10000 条

  @Cacheable(value = "sql_local", key = "T(...).md5(#question)")
  public SqlGenResult generate(String question) { ... }

层2：Redis 分布式缓存（跨实例共享）
  Key：sql:gen:{md5(question)}
  TTL：30 分钟
  当本地未命中，查 Redis；Redis 未命中，调 LLM
  调用后 set 回 Redis + Caffeine

层3：异步并行（虚拟线程）
  spring.threads.virtual.enabled: true

  @Async
  public CompletableFuture<SqlGenResult> generateAsync(...)

  并发调度多个独立查询（如多表关联场景）

层4：连接池调优
  Druid:
    initial-size: 10
    min-idle: 10
    max-active: 100        # 单实例最大并发查询数
    max-wait: 3000         # 拿不到连接 3 秒超时
    keep-alive: true

  HikariCP（如果用 HikariCP）:
    maximum-pool-size: 100
    connection-timeout: 3000

场景估算：
  目标：1000 QPS
  缓存命中率 70% --> LLM 实际 QPS = 300
  LLM 5 并发 + 单实例 --> 部署 60 个实例（虚拟线程下单实例可顶 5）
  实际：每实例 50 并发，部署 6 个实例足够
```

#### Day46 限流 + 熔断（用 Sentinel）

```
限流维度：

  全局：100 QPS（保护 LLM API 配额）
  单用户：10 QPS（防滥用）
  单 IP：30 QPS（防爬虫）

  // 复用 demo-gateway-sentinel 的能力
  @SentinelResource(value = "ai_sql_generate",
                    blockHandler = "handleBlock")
  public SqlGenResult generate(String question) { ... }

  public SqlGenResult handleBlock(String question, BlockException e) {
      return SqlGenResult.rateLimit("当前查询繁忙，请稍后再试");
  }

熔断：
  LLM 接口失败率 > 50% --> 熔断 30 秒
  期间所有请求直接返回兜底文案
  避免雪崩（LLM 挂了，整个服务不可用）

降级策略：
  熔断时返回："抱歉 AI 助手暂时不可用，请直接联系 DBA"
```

#### Day47 审计 + 异常处理

```
审计日志（必须，合规要求）：

  字段：
    request_id    全链路追踪 ID
    user_id       用户
    question      原始问题
    sql           生成的 SQL
    tables        涉及的表
    row_count     返回行数
    cost_ms       耗时
    success       是否成功
    error_msg     错误信息
    ip            来源 IP

  异步落库：
    - 业务线程不直接写 DB
    - 通过 Kafka --> 消费者批量写 audit_log 表
    - 不影响主流程性能

异常处理矩阵：

  异常类型              | 处理策略             | 用户提示
  ─────────────────────────────────────────────────────────────────
  LLM 超时              | 重试 1 次            | "AI 思考超时，重试中"
  LLM 配额不足          | 熔断 + 兜底           | "服务繁忙，稍后再试"
  SQL 解析失败          | 不重试，让 LLM 重生成 | "抱歉，请换个表达方式"
  SQL 含禁止操作        | 直接拒绝 + 审计       | "仅支持查询操作"
  SQL 执行超时          | DB 强制 cancel        | "查询时间过长，请缩小范围"
  连接池满              | 502 + Retry-After     | "服务繁忙，请稍后"
  结果集过大            | 截断 + 提示           | "结果较大，已显示前 1000 条"

  全局异常处理器（@ControllerAdvice）统一收口
```

## 法律 / 合规警示

```
⚠️ 爬虫法律红线：

  □ 不爬个人隐私（手机号、身份证）
  □ 不爬付费内容（绕过 paywall 是侵权）
  □ 不绕过显著的技术保护措施（破解登录是非法）
  □ 不高频请求（影响服务器正常运营 = 破坏计算机系统罪）
  □ 不存储后商业化转售（数据资产权归原平台）

合规建议：
  □ 严格遵守 robots.txt
  □ QPS 控制在 1 以内
  □ 仅采集公开热榜数据，不爬详情/评论
  □ 数据用于个人学习/内部分析，不对外提供
  □ 商用前找律师咨询

参考案例：
  - 大众点评诉百度案（爬数据展示构成不正当竞争）
  - 微博诉脉脉案（爬用户关系链构成不正当竞争）
  - 头条诉百家号案（爬正文+绕反爬构成侵权）
```
